# Database and Seed — Plan

The single source of truth for schema, RLS, and seed data. Every other feature either reads or writes these tables.

## Artefacts

```
backend/db/
├── migrations/
│   └── 0001_init.sql       # tables, RLS, create_order RPC, storage bucket
├── seed.sql                # canonical seed (generated by the script below)
├── create_admin.sql        # idempotent admin user bootstrap
└── README.md               # how to apply
```

## Tables (public schema)

- **`products`** — catalog; `id uuid pk`, `slug text unique`, `name`, `description`, `price_cents`, `currency`, `image_url`, `stock`, `is_active`, `created_at`, `updated_at`. `updated_at` maintained by a trigger.
- **`orders`** — one row per guest checkout; `email`, `shipping_name`, `shipping_address jsonb`, `subtotal_cents`, `total_cents`, `status text CHECK (status IN ('pending','paid','shipped','cancelled'))`.
- **`order_items`** — line items; FK to `orders` (cascade delete) and `products` (no cascade); `quantity`, `unit_price_cents` (snapshot at purchase).

## Indexes

- `products (is_active)` — supports the public filter.
- `products USING gin (name gin_trgm_ops)` — fuzzy name search (requires `pg_trgm`).
- `orders (status)`, `orders (email)`, `order_items (order_id)`.

## RLS

- `products` — SELECT policy `is_active = true` for anon; writes denied. FastAPI uses the service-role key and bypasses RLS.
- `orders`, `order_items` — no anon policies; backend-only access.

## Stored procedure

`public.create_order(p_email, p_shipping_name, p_shipping_address, p_items)` — heart of checkout:

1. Validates `items` non-empty and email shape.
2. Inserts a `pending` order.
3. For each item in `p_items`: `SELECT … FOR UPDATE` on the product (row lock), checks stock, inserts `order_items` with current price, decrements `products.stock`.
4. Flips the order to `paid` (payments mocked), returns `{ order_id, total_cents, status }`.
5. `GRANT EXECUTE` only to `service_role`.

## Storage

Migration `INSERT`s the `product-images` bucket (public=true) + SELECT policy making every object readable. Writes go through the service-role client in `services/storage.py`.

## Seeder

`backend/scripts/seed_shirts.py`:

- Fetches DummyJSON `mens-shirts` + `tops` categories.
- Transforms to `products` row shape.
- Deletes the old picsum-seeded slugs *only if* the `orders` table is empty (FK safety).
- Upserts rows via Supabase REST.
- Regenerates `backend/db/seed.sql` (idempotent upsert on `slug`).

Run:

```bash
cd backend
uv run python scripts/seed_shirts.py
```

## Depends on

- Nothing — base layer.

## Apply

Paste each file into Supabase SQL Editor, in order:

1. `backend/db/migrations/0001_init.sql`
2. `backend/db/seed.sql` (or run `seed_shirts.py` instead for real apparel)
3. `backend/db/create_admin.sql` (after editing email/password)

## Verify

```sql
select count(*) from products;             -- matches seed row count
select proname from pg_proc
 where proname = 'create_order';            -- exists
select * from storage.buckets
 where id = 'product-images';               -- exists, public=true
```
